Synergy Full Stack
Technical Specification

FastAPI backend with Server-Driven UI, Flutter mobile client, Supabase data layer, and Claude-powered activity generation engine.

Backend: FastAPI + Python 3.13
Mobile: Flutter 3.x (Dart)
Database: Supabase + Redis
AI: Google Genai API
Pattern: Server-Driven UI (SDUI)

Every technology choice is deliberate. Here's what we're using and why.

AI Layer

Deployment

One-command deploys, managed Supabase + Redis, free tier for MVP

Compile-safe, great async/network state handling, testable
💡
Key SDUI insight: Supabase's JSONB column type is central to this architecture. Activity configs, UI component trees, and theme tokens are all stored as JSONB — giving you schema flexibility without sacrificing query performance. You can index into JSONB fields and query them efficiently.

API Endpoint Specifications

All endpoints return JSON. Protected routes require Authorization: Bearer <token>.

Authentication — /api/v1/auth
POST/auth/registerCreate parent account
Body: email, password, full_name · Returns: user object + access_token
POST/auth/loginPassword login
Body: email, password · Returns: access_token (15min) + refresh_token (30d)
POST/auth/refreshRotate access token
Body: refresh_token · Returns: new access_token
Children — /api/v1/children
GET/children🔐 JWT
Returns all children belonging to authenticated parent. Includes active goal count.
POST/children🔐 JWT
Create child profile. Body: name, date_of_birth, interests[]
GET/children/{child_id}🔐 JWT
Full child profile with goals and recent session summary.
PUT/children/{child_id}/interests🔐 JWT
Update interests array. Triggers activity cache invalidation in Redis.
POST/children/{child_id}/goals🔐 JWT
Add a developmental goal. Body: domain, description, priority, set_by
Activities — /api/v1/activities
GET/activities/next/{child_id}🔐 JWT
The core endpoint. Returns a complete SDUI payload for the next activity. Flow:

1. Check Redis cache for a pre-generated activity for this child
2. If cache miss → call activity generator service → Claude API
3. Validate generated JSON against Pydantic schema
4. Store in DB, cache result, return to client

Query params: ?force_regenerate=true to bypass cache
GET/activities/{activity_id}🔐 JWT
Fetch a specific activity by ID. Used for replaying past activities.
GET/activities/ui-configNo auth required
Returns global app theme config. Called on app startup. Allows remote theme updates without app release.
Sessions — /api/v1/sessions
POST/sessions🔐 JWT
Save completed session. Body: child_id, activity_id, score, completed, parent_rating, notes, duration_seconds
GET/sessions/{child_id}/history🔐 JWT
Paginated session history. Query params: ?limit=20&offset=0&domain=communication
GET/sessions/{child_id}/progress🔐 JWT
Aggregated progress metrics per goal domain. Returns data for Flutter progress charts.

SDUI JSON Schema

The complete structure of an activity payload. This is the contract between server and Flutter client. The Pydantic models on the server and the Dart models on the client both implement this schema.

⚠️
Schema discipline is critical. Every field must be validated by Pydantic before the payload leaves the server. The Flutter client should never receive a malformed config — a validation error server-side is far better than a crash in a parent's hands mid-activity.

AI Activity Generation Engine

The generator is the brain of Synergy. It takes a child's profile and produces a validated SDUI activity payload using Claude's structured output via tool_use.

Generation Flow
GET /activities/next/{child_id} │ ▼ [1] Check Redis cache → HIT: return cached payload │ MISS ▼ [2] Load child context from Supabase • name, age, interests • active goals (ordered by priority) • last 5 sessions (avoid repetition) │ ▼ [3] Build structured prompt • System: therapy framework instructions • User: child profile + goal context • Tool: ActivityPayload JSON schema │ ▼ [4] Claude API call (tool_use mode) • Forces structured JSON output • No free-text hallucination risk │ ▼ [5] Validate response PASS: store in DB, cache 4hrs, return FAIL: log error, fall back to template activity
Activity Generator Service
Pythonapp/services/activity_generator.py
from google import genai from app.schemas.activity import ActivityPayload from app.models.child import Child import json client = genai.Client() async def generate_activity(child: Child, goals: list, recent_sessions: list) -> ActivityPayload: # Build rich context for Claude child_context = { "name": child.name, "age_years": child.age_in_years, "interests": child.interests, "active_goals": [ {"domain": g.domain, "description": g.description, "priority": g.priority} for g in goals ], "recent_game_types": [s.game_type for s in recent_sessions[-5:]] } response = await client.models.generate_content( model="gemini-3-flash-preview", max_tokens=2048, system="""You are a specialist in autism therapy and child development. Generate structured activity configurations for parents to use at home. Activities must be evidence-based (ABA / naturalistic teaching), 5-10 minutes long, and appropriate for the child's age and goals. Always vary game types from recent sessions. Never use clinical jargon in parent-facing text — keep instructions warm, simple, and encouraging.""", tools=[{ "name": "create_activity", "description": "Creates a structured SDUI activity payload", "input_schema": ActivityPayload.model_json_schema() }], tool_choice={"type": "tool", "name": "create_activity"}, messages=[{ "role": "user", "content": f"Generate a home activity for this child: {json.dumps(child_context)}" }] ) # Extract tool_use result and validate with Pydantic tool_result = next(b for b in response.text if b.type == "tool_use") return ActivityPayload.model_validate(tool_result.input) # Raises on bad output
ℹ️
Why tool_use instead of prompting for JSON? Claude's tool_use mode guarantees structured output — it cannot deviate from the schema. Asking Claude to "respond in JSON" in a regular message still allows freeform responses. For a healthcare-adjacent product, structured output is non-negotiable.

Auth & Security

Pythonapp/core/security.py
from datetime import datetime, timedelta from jose import jwt, JWTError from passlib.context import CryptContext from app.core.config import settings pwd_context = CryptContext(schemes=["bcrypt"], deprecated="auto") def create_access_token(user_id: str) -> str: payload = { "sub": user_id, "exp": datetime.utcnow() + timedelta(minutes=15), "type": "access" } return jwt.encode(payload, settings.SECRET_KEY, algorithm="HS256") def create_refresh_token(user_id: str) -> str: payload = { "sub": user_id, "exp": datetime.utcnow() + timedelta(days=30), "type": "refresh" } return jwt.encode(payload, settings.SECRET_KEY, algorithm="HS256") # FastAPI dependency — inject into any protected route async def get_current_user(token: str = Depends(oauth2_scheme), db = Depends(get_db)): try: payload = jwt.decode(token, settings.SECRET_KEY, algorithms=["HS256"]) user_id: str = payload.get("sub") if payload.get("type") != "access": raise credentials_exception except JWTError: raise credentials_exception return await db.get(User, user_id)
🔒
Child data is sensitive. Enforce row-level security: every DB query for child/session data must filter by parent_id = current_user.id. Never return child data for a parent who doesn't own it. This is the single most important security invariant in the entire codebase.

Flutter SDUI Engine Design

The engine is a pure JSON-to-Widget translator. It has no business logic — it only knows how to map component types to pre-built widgets.

Dartlib/sdui/component_registry.dart
/// Central registry mapping server component_type strings → Widget builders /// To add a new game: register it here + build the widget in /games/ typedef ComponentBuilder = Widget Function(Map<String, dynamic> config, SessionController session); class ComponentRegistry { static final Map<String, ComponentBuilder> _builders = { // ── Game widgets ────────────────────────────────────── 'matching_game': (config, session) => MatchingGame(config: config, session: session), 'sequencing_game': (config, session) => SequencingGame(config: config, session: session), 'naming_game': (config, session) => NamingGame(config: config, session: session), 'imitation_game': (config, session) => ImitationGame(config: config, session: session), // ── UI components ───────────────────────────────────── 'header': (config, _) => SduiHeader.fromJson(config), 'progress_bar': (config, _) => SduiProgressBar.fromJson(config), 'parent_prompt': (config, _) => SduiParentPrompt.fromJson(config), 'celebration': (config, _) => SduiCelebration.fromJson(config), }; static Widget build(String componentType, Map<String, dynamic> config, SessionController session) { final builder = _builders[componentType]; if (builder == null) { // Graceful degradation — never crash, show placeholder return UnknownComponentPlaceholder(type: componentType); } return builder(config, session); } }

Game Widget Registry

All games extend BaseGame and implement a standard interface. They receive server config as a Map and report completion via SessionController.

component_type Description Config Keys Therapy Domain
matching_game Drag items into category buckets items, categories, time_limit Cognitive / Classification
sequencing_game Arrange steps in correct order steps, shuffle, show_numbers Cognitive / Planning
naming_game Tap the image that matches a label items, distractors, audio_prompt Communication / Vocabulary
imitation_game Watch + tap to replay a gesture sequence sequence, delay_ms, repetitions Social / Imitation
choice_game Make a 2–4 option choice (Phase 2) prompt, options, correct_id Communication / Choice-making
emotion_game Match faces to emotion labels (Phase 2) emotions, difficulty Social / Emotional recognition

/// All game widgets must extend this abstract class abstract class BaseGame extends ConsumerStatefulWidget { final Map<String, dynamic> config; final SessionController session; const BaseGame({required this.config, required this.session, super.key}); } abstract class BaseGameState<T extends BaseGame> extends ConsumerState<T> { int correctCount = 0; int totalCount = 0; DateTime? startTime; @override void initState() { super.initState(); startTime = DateTime.now(); } /// Called when game is complete — reports to SessionController void completeGame() { final duration = DateTime.now().difference(startTime!).inSeconds; widget.session.recordResult( correct: correctCount, total: totalCount, durationSeconds: duration, ); } /// Called on each correct answer — plays celebration micro-animation void onCorrectAnswer() { correctCount++; // Trigger haptic + visual feedback } }

Step-by-Step Build Plan

5 phases, sequenced so the API and Flutter client can develop in parallel after Phase 1. Each phase ends with a working, testable milestone.

  • Define all Pydantic schemas — especially ActivityPayload
    This is the contract. Write it carefully — both the AI generator and the Flutter client are built around it. Add .model_json_schema() export for Claude tool_use integration.
    FastAPIAI

    Run flutter create synergy_app, install all pubspec dependencies, create the folder structure. Configure go_router with placeholder routes for auth, home, activity, and progress screens.
    Flutter

    Create ActivityPayload, ThemeConfig, GameConfig, etc. in Dart using json_serializable. These mirror the Pydantic models 1:1. This step unblocks all Flutter UI work.
    Flutter
